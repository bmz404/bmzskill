# Suno 技术方法论

证据等级：B 级方法论  
用途：供 `/bmz-ask` 回答 Suno、voice、cover、remix、extend、prompt、元标签、段落控制、编曲控制、男女对唱、瑕疵拼接、咬字、局部修改、下载导出、音频处理边界等问题。  
边界：Suno 等音乐生成平台更新频繁，按钮名称、入口位置、会员权益和模型能力可能变化。回答用户时，不要把旧界面路径写成绝对步骤；应以“功能方向 + 当前界面核对”为主。

## 1. Suno 在 bmzskill 里的定位

Suno 不是一键许愿机，而是高效率音乐试错工具。

bmzskill 使用 Suno 的核心目的不是“直接做最终成品”，而是：

1. 快速验证歌词、曲风、人声和编曲方向。
2. 快速生成多个 demo 版本，筛掉明显不成立的方向。
3. 用短视频主打段测试市场反馈。
4. 当短视频数据成立后，再进入完整版、正式编曲、真人演唱、混音、母带、发行或版权交易。

因此，Suno 生成结果默认是 demo。不要把 demo 当成最终母带，也不要因为某一版 demo 听感不错，就跳过短视频测试。

## 2. 使用前先明确目标

进入生成前，先判断用户要解决的问题属于哪一类：

- 做 8 句短视频 hook。
- 做完整 demo。
- 做纯音乐氛围段。
- 固定某个 voice。
- 用已有片段 extend。
- 用 cover 改曲风或改人声。
- 修正唱错字、咬字、节奏、段落不完整。
- 延长前奏或补尾奏。
- 下载、导出、分轨或后期处理。

不同目标对应不同操作。不要所有问题都回答“重新生成”。

## 3. 推荐基础流程

最稳的流程是：

1. 先确定主打段用途：短视频测试 / 完整 demo / 发行前样稿。
2. 准备歌词：短视频阶段优先 8 句 hook；完整歌再选结构。
3. 准备 style prompt：曲风、人声、情绪、tempo、主要编曲质感。
4. 准备 exclude：不想要的风格、质感、人声问题。
5. 设置高级参数：只保留必要项目，避免乱调。
6. 一次生成多个版本。
7. 先选“方向正确”的版本，不急着选“局部最完美”的版本。
8. 用 extend / edit / cover / 重写歌词局部解决问题。
9. 导出后再做短视频测试。

判断版本时按优先级：

1. 曲风方向是否对。
2. 第一口人声是否抓人。
3. 第一二句歌词是否唱清楚。
4. 旋律记忆点是否靠前。
5. 编曲是否让歌词站出来。
6. 是否有明显不可修复的问题。

## 4. 简单模式适合什么

简单模式适合快速找灵感，不适合精确控制。

适合：

- 用户只有一句主题或情绪。
- 想快速听几个大方向。
- 还不确定曲风。
- 想从生成结果里找旋律灵感。

不适合：

- 已经有明确歌词。
- 想控制段落结构。
- 想固定人声。
- 想做短视频主打段精确测试。
- 想控制某几句必须怎么唱。

如果用户已经有歌词、曲风或对标，优先建议进入自定义模式。

## 5. 自定义模式适合什么

自定义模式适合需要控制歌词、段落、曲风和人声的场景。

基本输入包括：

- Lyrics：歌词、段落标签、编曲元标签。
- Style Prompt：曲风、人声、情绪、tempo、制作质感。
- Exclude：排除不想要的结果。
- Advanced：只使用必要参数。

短视频 hook 测试时，不要为了完整结构强行写 intro、verse、bridge、outro。8 句主打段默认可以写：

```text
[Hook]
[stripped down first four lines, close vocal only, soft drums enter on line 5, layered harmonies on last line]
歌词第1句
歌词第2句
歌词第3句
歌词第4句
歌词第5句
歌词第6句
歌词第7句
歌词第8句
```

## 6. 纯音乐生成

纯音乐适合：

- 情绪 BGM。
- 氛围短视频。
- DJ / 卡点 / 运动 / 游戏 / 换装场景。
- 后续给人声留空间的伴奏草稿。
- 影视感、国风、赛博、钢琴、吉他等氛围素材。

纯音乐 prompt 要更重视：

- 使用场景。
- 节奏强度。
- 乐器主次。
- 情绪曲线。
- 是否需要 drop。
- 是否需要循环感。

不要只写“sad instrumental”或“beautiful music”。这类提示太空，容易生成没有使用场景的背景音乐。

## 7. Style Prompt 写法

有效 prompt 不靠堆词，而靠边界清晰。

推荐结构：

```text
{language} {genre}, {emotion}, {tempo feel}, {main instruments}, {vocal tone}, {production texture}
```

示例：

```text
Mandarin emo pop with alternative R&B influence, intimate and bittersweet, mid tempo, soft drums, warm bass, sparse piano, vulnerable young male vocal, clear Mandarin pronunciation, clean modern production
```

不要同时塞太多互相冲突的曲风。例如：

```text
R&B, rock, folk, EDM, trap, jazz, cinematic, cute, dark, happy, sad
```

这会让模型不知道谁是主方向。

## 8. Prompt 的四层控制

### 曲风层

告诉模型“这是什么歌”：

- Mandarin pop
- Chinese R&B
- emo pop
- melodic rap
- acoustic pop
- Chinese style R&B
- city pop
- pop rock
- K-pop inspired urban pop

### 情绪层

告诉模型“这首歌让人怎么感受”：

- intimate
- bittersweet
- lonely
- nostalgic
- restrained
- warm
- heartbroken
- hopeful

### 编曲层

告诉模型“音乐怎么搭起来”：

- acoustic guitar
- soft piano
- warm bass
- muted drums
- 808 drums
- airy pads
- electric piano
- clean guitar
- subtle strings

只写用户能确认的方向。不要凭空写具体乐器。

### 人声层

告诉模型“谁在唱、怎么唱”：

- soft male vocal
- breathy female vocal
- youthful male vocal
- clear Mandarin pronunciation
- intimate vocal delivery
- emotional but restrained
- slight R&B phrasing

## 9. Exclude 排除词

Exclude 的作用是减少明显不想要的结果，不是把模型锁死。

常见可排除：

```text
no heavy metal, no EDM drop, no choir, no opera, no distorted vocal, no robotic vocal, no messy pronunciation, no childish melody, no over-compressed mix
```

如果目标是短视频情歌，可以排除：

```text
no upbeat dance pop, no comedy tone, no dramatic orchestral intro, no long instrumental intro
```

如果目标是清晰中文咬字，可以排除：

```text
no mumbling, no unclear Mandarin, no exaggerated vibrato
```

不要写太多排除词。排除词太多，可能让结果变得僵硬或失焦。

## 10. Advanced options

bmzskill 默认只使用两个高级项：

- weirdness
- style influence

输出时必须给具体数值区间，不写 low / medium / high。

区间不要超过 20 个点。例如：

```text
weirdness: 15-30
style influence: 65-80
```

使用原则：

- 想要更稳、更像参考方向：提高 style influence，降低 weirdness。
- 想要更多意外灵感：略提高 weirdness。
- 短视频测歌阶段不建议 weirdness 太高，因为第一目标是可传播、可理解、可复现。

## 11. Lyrics 区域的段落标签与元标签

元标签是写在 Lyrics 区域里的中括号指令。最基础的元标签是段落标签，例如 `[Verse]`、`[Chorus]`、`[Bridge]`；更进一步，也可以写局部编曲、和声、空拍、口白、效果和演唱分配。

Style Prompt 和元标签的分工不同：

- Style Prompt：规定整首歌的宏观方向，例如曲风、情绪、人声、速度、制作质感。
- 元标签：规定某一个局部应该怎么发生，例如这一句谁唱、这里空几拍、这里加入和声、这里用回声收尾。

所以 Style Prompt 像“整首歌的方向盘”，元标签像“局部转弯时的路标”。元标签不能保证 100% 执行，但它能显著提高局部控制概率。

### 常用段落标签

段落标签用于帮助模型理解结构，但不能保证 100% 精确执行。

```text
[Intro]
[Verse]
[Pre-Chorus]
[Chorus]
[Hook]
[Bridge]
[Outro]
```

短视频主打段优先 `[Hook]`。

完整歌可以根据结构选择：

```text
[Verse 1]
[Pre-Chorus]
[Chorus]
[Verse 2]
[Pre-Chorus]
[Chorus]
[Bridge]
[Final Chorus]
[Outro]
```

不要为了显得完整而乱加段落。段落越多，模型越可能偏移。

## 12. 用元标签控制断句、呼吸和局部表现

### 1. 控制断句和呼吸感

如果歌曲排得太满、没有气口，不要只在 Style Prompt 里写“breathy”或“more space”。更有效的方法是在 Lyrics 中需要停顿的位置插入局部指令，让模型知道这里不是继续塞歌词，而是用乐器填充。

示例：

```text
[Hook]
你说随便我
像同一扇门又推开我
[Instrumental Break: short guitar fill, one bar]
我听了很多遍
还是听不到你的挽留
```

可用方向：

- `[Instrumental Break: short piano fill]`
- `[Instrumental Break: one bar guitar fill]`
- `[Short Pause: drums and bass continue]`
- `[Breath: vocal leaves space before next line]`

断句元标签适合用在句尾、情绪转折点、歌词密度过高的位置。不要每一句都塞空拍，否则主打段会被拖慢。

### 2. 做和声与人声层次

如果副歌只有一轨人声，听起来太薄，可以在副歌元标签里明确加入和声或多层人声。

示例：

```text
[Hook]
[lead vocal with soft layered harmonies, backing vocals enter on line endings]
我不是放不下你
是还没学会告别
```

可用方向：

- `[soft layered harmonies]`
- `[backing vocals answer the lead vocal]`
- `[double-tracked chorus vocal]`
- `[wide background harmonies in the final lines]`

注意：和声会增加厚度，也可能盖住歌词。短视频主打段优先保证第一句歌词清楚。

### 3. 加 Ad-libs

Ad-libs 适合让段落更有活人感、律动感或情绪爆点，但不要滥用。

示例：

```text
你又说 说随便我
[Ad-Libs: hey]
我听了很多遍
[Ad-Libs: oh no]
```

常见写法：

- `[Ad-Libs: hey hey]`
- `[Ad-Libs: oh baby]`
- `[Ad-Libs: yeah]`
- `[Ad-Libs: ah]`

中文情歌里 Ad-libs 要克制。太多会显得油、乱、廉价。

### 4. 加 Spoken Word

如果想制造更私密、更 emo 的氛围，可以让某个位置出现近似说话的片段。

示例：

```text
[Spoken Word: 我其实已经知道答案了]
你还是没有回头
```

适合：

- 分手独白
- 低声告白
- 转场前的心理活动
- 结尾留白

不适合：

- 强旋律副歌第一句
- 需要快速抓人的前 2 秒
- 已经很满的旋律说唱段落

### 5. 做效果与尾奏

可以用元标签提示局部音效，例如回声、渐弱、失真、空间感。

示例：

```text
[Outro]
[Echo: let me go, let me go]
[Fade Out: soft reverb tail]
```

常见方向：

- `[Echo: ...]`
- `[Fade Out]`
- `[Reverb tail]`
- `[Distant vocal]`
- `[Filtered vocal]`

效果标签适合尾奏、桥段、转场，不建议堆在第一句主打歌词上。

## 13. 用元标签控制编曲推进

编曲元标签写在 Lyrics 区域，用来提示某一段的编曲状态。它最适合用来安排“乐器什么时候进来、什么时候变厚、什么时候收掉”。

简单示例：

```text
[Hook]
[soft drums enter, warm bass, intimate vocal, subtle piano]
我不是放不下你，是还没学会告别
我们的世界，擦肩以后断裂
```

常见标签方向：

- `[soft piano intro]`
- `[acoustic guitar enters]`
- `[drums enter softly]`
- `[bass becomes warmer]`
- `[vocal becomes more emotional]`
- `[drop with muted drums]`
- `[stripped down]`
- `[final chorus, fuller arrangement]`

不要写太复杂的制作术语。模型更容易理解清晰的情绪和乐器状态。

更完整的编曲推进示例：

```text
[Intro]
[acoustic guitar solo, intimate room tone]

[Verse 1]
[piano enters softly, vocal close and restrained]

[Pre-Chorus]
[warm bass enters, tension slowly builds]

[Hook]
[full band enters, soft drums, warm bass, piano and acoustic guitar, emotional lead vocal]
```

核心原则：

- 前奏只给一个清晰主乐器，不要一上来塞满。
- 主歌加一点支撑，但要让歌词站出来。
- 预副歌负责蓄力。
- 副歌再让主要乐器一起进入。

元标签不是越多越专业。每一段只写当前最重要的 1-3 个变化。

## 14. 男女对唱段落分配

如果要做男女对唱，不要只在 Style Prompt 里写 duet。更稳的做法是同时在 Style Prompt 和 Lyrics 区域定义角色。

### 第一步：Style Prompt 说明这是对唱

示例：

```text
Mandarin pop duet with male and female vocals, clear call-and-response structure, emotional dialogue, both singers harmonize with each other in the chorus, clean modern production
```

### 第二步：Lyrics 顶部定义角色

示例：

```text
[Duet: John = male vocals, Mini = female vocals, Both = John and Mini sing together]
```

这里的名字只是为了让模型更好分配角色，不需要是真实姓名。

### 第三步：每句歌词前标清是谁唱

示例：

```text
[Duet: John = male vocals, Mini = female vocals, Both = John and Mini sing together]

[Verse 1]
[John] 你说随便我
[Mini] 可我听见的不是自由
[John] 我退到门口
[Mini] 你却没有一句挽留

[Hook]
[Both] 我们怎么两清
[John] 怎么做回甲乙丙丁
[Mini] 你在我的世界
[Both] 早就不是普通姓名
```

Advanced 里的性别选项不要和 prompt 冲突。做男女对唱时，通常不要再额外选择单一性别，否则可能造成输出混乱。

注意：对唱分配仍然是概率执行。若最后两句或局部角色错位，不要急着推翻整首，先多生成几版，选角色分配最接近的版本继续处理。

## 15. Voice 固定音色

voice 的作用是提高人声一致性，不是保证每次完全一样。

适合：

- 同一账号需要稳定声音记忆。
- 同一首歌多次迭代，希望音色接近。
- 测试男女声版本。
- 测试同一 hook 的不同唱法。

使用 voice 时要注意：

1. 不要让 voice 和曲风冲突。甜美女声不一定适合硬核旋律说唱，低沉男声不一定适合轻快甜歌。
2. 不要只靠 voice 解决咬字问题。歌词行长、断句、多音字、音区更重要。
3. 不同 key 会改变同一个 voice 的听感。
4. 同一 voice 也可能因为 EQ、混音色彩、伴奏密度不同而听上去不一样。

## 16. Voice / Cover 音色不一致排查

当用户说“同一个 voice 每次 cover 出来的音色还是有区别”，按以下顺序排查：

### 1. 先判断是不是 EQ 差异

同一首歌不同版本听起来不一样，可能不是 voice 变了，而是混音明暗不同。

- 太闷：适当增加高频或空气感。
- 太刺：压一点高频或齿音区。
- 太薄：检查中低频是否不足。
- 太糊：削掉多余低频和低中频。

如果 EQ 一调就接近，说明主要不是音色漂移。

### 2. 再判断 key 是否不同

不同歌曲、不同 key 会让同一个 voice 表现不同。

- key 太高：可能更尖、更紧、更亮。
- key 太低：可能更厚、更暗、更闷。
- 副歌长期卡在高音区：声音会变得不像原来。

### 3. 再看歌词密度

字太密、句子太长、换气点太少，会让 voice 不稳定。

解决方式：

- 缩短句子。
- 增加自然断句。
- 把重要词提前。
- 减少连续拗口字。

### 4. 最后才换 voice

如果 EQ、key、歌词密度都排查过，仍然明显不稳定，再考虑换 voice 或重新生成。

## 17. 瑕疵版本拼接修复

Suno 生成结果有瑕疵是正常现象，尤其是中文歌。不要默认追求“一次生成完美版”，更稳的思路是先降低瑕疵概率，再把多个可用版本拼成一个更好的版本。

### 一版音色好、一版旋律好怎么办

先分清用户到底要解决什么问题：

- 如果用户想把两版里的可用句子拼成一个成品，这是“瑕疵版本拼接修复”，走下载、分离、挑段、Crossfade、EQ、统一音量。
- 如果用户的目标是“用第一版的人声音色唱第二版的旋律”，这不是拼接问题，而是 Voice + Cover 问题。

当用户说“第一版音色好，第二版旋律好，我想让第一版音色唱第二版旋律”时，优先回答：

1. 先使用 Suno Voice 功能，把第一版里满意的人声音色存入 Voice 音色库。
2. 再选择第二版作为旋律/歌曲参考。
3. 用刚存下的 Voice 去 Cover 第二版，让第二版旋律尽量由第一版的人声音色重新演唱。
4. 如果输出后音色仍有明暗差异，先判断是不是 EQ、key、伴奏密度造成的听感差异。
5. 如果局部咬字或高音仍不理想，再多生成几版，必要时才进入后期拼接修复。

对用户的简洁说法：

```text
你这个需求不是把两首歌剪在一起，而是“用 A 音色重唱 B 旋律”。更合适的做法是：先把第一版满意的人声音色用 Suno Voice 存进音色库，然后用这个 Voice 去 Cover 第二版。这样目标是让第二版的旋律框架保留下来，同时尽量换成第一版的人声音色。
```

注意：Voice + Cover 也不是 100% 精准复刻。它能提高“第一版音色唱第二版旋律”的概率，但如果第二版 key 太高、歌词密度太大、伴奏太满，音色仍可能变化。

### 先降低可控错误

生成前先检查：

- Style Prompt 和元标签是否冲突。
- Lyrics 里的段落标签是否过多。
- 是否有多音字、生僻字、拗口连续音。
- 句子是否太长，导致咬字和换气困难。
- voice、key、曲风是否匹配。

### 多版本拼接流程

如果方向成立，但每一版都有局部小瑕疵，可以这样处理：

1. 用同一个 voice 多生成或 cover 几个版本。
2. 下载每个版本。
3. 用 UVR5 或同类工具做人声与伴奏分离。
4. 把可用的人声轨导入宿主软件。
5. 从不同版本里挑最好的句子或短段落拼接。
6. 拼接处如果有卡顿，用 Crossfade 做过渡。
7. 如果不同版本音色略有明暗差异，用 EQ 做接近处理。
8. 最后统一音量，再导出完整版本。

这个方法适合“方向好，但局部不完美”的 demo，不适合拯救旋律、歌词、曲风整体不成立的作品。

### EQ 修补的判断

有时候听起来像“音色变了”，其实只是 EQ 不同：

- 这一版亮，另一版闷：先调 EQ。
- 这一版厚，另一版薄：看中低频和低频支撑。
- 齿音、鼻音、空气感不同：先做局部频段处理。

如果 EQ 一调就接近，说明主要问题不是 voice 漂移。

## 18. Cover / Remix 使用原则

cover 更像“基于已有素材重新演绎”，不是万能修复。

适合：

- 用户有自己哼唱的旋律动机。
- 用户想保留某段旋律轮廓，换曲风或换人声。
- 用户想测试同一 hook 的不同音色版本。
- 用户有一段满意 demo，想生成更接近目标质感的版本。

不适合：

- 原素材本身节奏、旋律、咬字都很乱。
- 想绕过版权直接复刻现存歌曲。
- 想 100% 保留原唱神态。
- 想把干声硬塞回复杂编曲并自动完全贴合。

如果用户想替换某段高音、戏腔或长音，优先建议：

1. 判断这段是否适合重新生成整段，而不是只替换一个音。
2. 如果是合唱或高音长音，干声直接塞回去容易失去空间感。
3. 需要考虑混响、延迟、EQ、压缩、音量包络和段落长度。
4. 时间对不上时，先对齐节拍和句尾落点，再考虑音色。

### 用 Cover 做风格重塑

如果用户想把一首已有旋律改成全新风格，核心是只给模型“人声/旋律骨架”，不要把原伴奏一起喂进去。否则模型会继续参考原伴奏，风格反差很难拉开。

推荐流程：

1. 只准备 vocal 音频，不带伴奏。
2. 上传音频后选择 Cover，确认 Lyrics 区域上方已经挂载该音频。
3. 填入授权范围内可使用的歌词，或填入用户自己的改写歌词。
4. Style Prompt 写清楚新曲风、新人声、新制作质感。
5. 追求“有反差但不毁歌”：新风格要让人好奇，但不能让原旋律失去可听性。
6. Advanced 参数不要拉到极端，避免乱码、音质变差或结构失控。

如果界面里没有看到 Audio Influence，常见原因是音频只是被上传到了工作区，但没有真正以 Cover 方式挂载到本次生成里。此时先回到 Lyrics 区域上方检查音频是否已经进入当前任务。

### Cover / Remix 参数原则

Cover 类任务常见参数方向：

- Variation：控制自由发挥程度。想保留旋律轮廓时不要太高。
- Style Influence：控制提示词遵循程度。想明显换风格时可以适当提高。
- Audio Influence：控制上传音频参考程度。默认中低区间通常能保留基本旋律框架；想更还原旋律可提高，但不要拉到极端。

注意：不同平台版本的参数命名和位置可能变化。回答用户时不要把旧界面路径写成绝对步骤，应以“功能方向 + 当前界面核对”为主。

### 音调问题

Cover 输出会受到上传音频音调影响。如果生成结果整体太高、太低、主歌副歌音区不自然，先不要只改 prompt，要检查源音频的 key。

处理方式：

- 源音频太高：先降调再上传。
- 源音频太低：先升调再上传。
- 主歌和副歌音区跨度异常：优先换更合适的源音频片段，或分段处理。

可以用常见剪辑或音频软件做简单升降调，再导出重新上传。

### 版权边界

Cover / Remix 涉及现存作品时，必须先判断授权风险。不能把“能生成”理解成“能公开发布或商用”。如果没有明确授权，只能作为内部测试和学习方向，不应直接发行、商用或伪装原创。

## 19. Extend 使用原则

extend 适合续写一个已经成立的片段。

适合：

- 前 8-15 秒很好，想继续往后发展。
- 副歌不错，想补前奏、主歌或尾奏。
- 想延长情绪而不重做整体方向。

不适合：

- 原片段曲风已经跑偏。
- 人声咬字严重错误。
- 节奏不稳。
- 旋律不成立。

使用 extend 前先问：

- 续写点是不是段落边界？
- 前一段有没有明确 tempo 和和声气质？
- 想续写的是同情绪、升情绪，还是转段？

## 20. 前奏太短怎么办

如果用户很喜欢当前版本，但前奏太短，不要建议一口气重做整首。

可选路径：

1. 用 extend 从开头或前奏附近延展，生成更长铺垫。
2. 重新生成一版“longer intro / soft instrumental intro”，但保留原 style prompt 和歌词。
3. 如果已有可用伴奏段，可以在后期复制类似氛围段，但要注意节拍和和声是否自然。
4. 如果只是短视频测试，不一定需要长前奏。短视频通常更需要快速进入有效主打段。

回答时不要承诺能听音频判断“该复制哪一段”。如果没有可靠音频分析，只能给通用处理路径。

## 21. 迭代生成

迭代时不要一次改太多变量。

每轮只改一类：

- 改歌词。
- 改曲风。
- 改 vocal。
- 改 tempo。
- 改编曲密度。
- 改段落结构。

如果同时改歌词、曲风、voice、tempo 和高级参数，就不知道下一版变好或变差是因为什么。

建议记录：

- 版本编号。
- 改了什么。
- 哪一句更好。
- 哪一句更差。
- 是否适合短视频测试。

## 22. 上传 / 录制音频

上传或录制音频适合提供：

- 哼唱旋律动机。
- 吉他/钢琴和弦走向。
- 人声情绪方向。
- 节奏律动参考。
- 一段已有 demo。

注意：

- 上传素材越乱，生成结果越难稳定。
- 哼唱跑调不一定致命，但节奏和旋律轮廓要清楚。
- 如果只是想给模型“灵感”，素材可以短；如果想让模型保留结构，素材要更明确。
- 涉及现存歌曲、受保护歌词、受保护人声时，必须先判断授权风险。

## 23. 人声加伴奏 / 伴奏加人声

### 人声加伴奏

适合用户已经有人声旋律或清唱片段，希望生成伴奏。

重点：

- 人声音准和节奏越清楚，伴奏越容易贴合。
- prompt 要说明曲风和编曲方向。
- 如果人声节奏自由，伴奏可能难以稳定进入。

### 伴奏加人声

适合已有伴奏或音乐氛围，希望生成 vocal。

重点：

- 歌词行长要适合伴奏速度。
- 人声 prompt 要清楚。
- 如果伴奏太满，人声容易被挤压。
- 如果伴奏没有明确段落，生成的人声也可能结构混乱。

## 24. 编辑器与局部修改

局部修改适合解决小问题，不适合救整首方向。

适合：

- 某一句唱错。
- 某个字咬不清。
- 某一句情绪不够。
- 某一段需要重唱。
- 想替换局部旋律。

不适合：

- 整首曲风不对。
- 主旋律不成立。
- 人声完全不适合。
- 结构混乱。

局部修改前先确认：

- 修改范围不要太短，否则上下文不够。
- 修改范围不要太长，否则等于重做。
- 修改后的歌词节奏要能接回前后文。
- 如果前后音区差太大，接缝会明显。

如果用户整体喜欢这一版，只是某一句咬字不准、唱错字或局部有瑕疵，不要建议推翻重做。更稳的处理是：

1. 将生成的多个版本全部下载。
2. 做音频分离，至少分离出人声轨。
3. 选最喜欢的一版作为主版本。
4. 在宿主软件里，把主版本放在主轨道。
5. 从其他版本的人声轨里，找到同一句里咬字正确、情绪也接近的那一小段。
6. 把这句正确的人声拼进主版本对应位置。
7. 拼接处用短 Crossfade，避免突然跳、卡顿或音色断裂。

这套方法适合“整体满意，只修某一句”的情况；如果整段旋律、节奏或情绪都不成立，才考虑重新生成或改 prompt。

## 25. 裁剪功能

裁剪适合：

- 去掉多余前奏。
- 截出短视频主打段。
- 去掉坏尾巴。
- 保留最抓人的 20-30 秒。

裁剪时优先保留：

- 第一口人声。
- 第一二句金句。
- 旋律记忆点。
- 情绪爆发点。

不要只按时间裁。要按音乐句子裁，否则容易切断呼吸、和弦或节拍。

## 26. BPM 与速度

BPM 数字不等于传播体感。

同样 146 BPM，可能听起来像中速，也可能像快歌，取决于鼓组、歌词密度和律动。

调整 BPM 时要注意：

- 只想让歌更紧凑，不一定要大幅变速。
- 人声变速会影响音色。
- 速度变快可能让咬字更差。
- 速度变慢可能让短视频前 5 秒承接变弱。

短视频主打段更重要的是：第一句进入是否快、记忆点是否靠前、歌词是否清楚。

## 27. 渐出与尾奏

渐出适合：

- 氛围歌。
- demo 片段。
- 短视频结尾。
- 不需要强结尾的情绪流歌曲。

不适合：

- 需要明确 ending 的歌曲。
- 强情绪爆发后需要落点的副歌。
- 需要观众记住最后一句的主打段。

如果主打段是为了短视频传播，结尾最好有一句态度、疑问或余味，不要突然断掉。

## 28. 母带提升

母带提升只能改善听感，不会改变歌曲本质。

适合：

- demo 整体已经成立，只是音量、亮度、厚度不够。
- 想让导出版本更适合试听。
- 想减少明显毛糙感。

不适合：

- 旋律不好。
- 歌词不清楚。
- 人声不适合。
- 编曲方向错。

不要用母带处理掩盖创作问题。

## 29. 下载、导出与素材管理

导出时至少保留：

- 完整音频。
- 主打段音频。
- 歌词文本。
- style prompt。
- exclude。
- voice 名称或描述。
- 生成版本编号。
- 使用日期。
- 适配的短视频模板。

如果后续要发行、版权交易或团队协作，更要保留：

- WAV 或高质量音频。
- 分轨或可用素材。
- 歌词版本。
- 创作说明。
- 测试数据。
- 授权和来源记录。

## 30. 音频分离与后期边界

音频分离可以帮助拆出人声、伴奏或参考素材，但分离结果不一定干净。

适合：

- 拿到人声参考。
- 做简单后期处理。
- 判断某一版人声是否能保留。
- 给后续制作提供素材。

风险：

- 分离后会有水声、毛边、相位问题。
- 分离的人声直接使用可能听感粗糙。
- 涉及现存作品时仍然有版权风险。

bmzskill V1 不把自动音频识别作为强判断来源。不要声称能准确识别 BPM、调性、和弦或旋律走向。

## 31. 灵感功能与随机探索

灵感类功能适合在没有方向时打开思路。

适合：

- 找曲风。
- 找旋律轮廓。
- 找 vocal 质感。
- 找编曲氛围。
- 找意外的副歌走向。

但灵感不是最终方案。拿到灵感后要回到 bmzskill 判断：

- 是否适配听众。
- 是否适配画面。
- 是否适配歌词。
- 是否适合短视频前 5 秒。
- 是否能形成可复用模板。

## 32. 咬字问题

唱错字、咬字糊、中文不清楚时，处理顺序是：

1. 改歌词行长。
2. 改断句。
3. 避免多音字和拗口连续音。
4. 把重要词放在更容易唱的位置。
5. 在 prompt 加 clear Mandarin pronunciation。
6. 降低句子密度。
7. 换更适合中文咬字的 voice。
8. 局部重生成或多版本同句替换。

如果用户已经有一版整体满意，只是某句咬字瑕疵，优先建议：

1. 下载多个生成版本。
2. 音频分离出人声轨。
3. 选最喜欢的一版作为主版本。
4. 从其他版本人声轨里挑咬字正确的同一句。
5. 在宿主软件里拼到主版本对应位置。
6. 拼接处用短 Crossfade，避免突然跳。

不要只靠“再生成一次”。如果歌词本身太挤，再生成也容易继续错。

## 33. 旋律统一

想让旋律更统一，重点不是写“旋律统一”，而是减少模型自由发挥空间。

方法：

- 缩短测试段。
- 保持歌词行长接近。
- 保持押韵和节奏相近。
- 不在同一段里塞多个曲风。
- 使用稳定 voice。
- 降低 weirdness。
- 提高 style influence。
- 用 cover 或 extend 保留已成立片段。

## 34. 全局统一

完整歌常见问题是：第一段好听，后面跑偏。

解决方式：

- 先确定主打段，再反推主歌和桥段。
- 每段只承担一个功能。
- 不要每段都写新情绪。
- 让 verse、pre-chorus、chorus 的歌词密度有区别。
- 最终副歌可以加厚，但不要突然换歌。

## 35. 编曲变动控制

如果用户说“编曲突然变了”，先判断是好变化还是坏变化。

好变化：

- 副歌进入时更饱满。
- 情绪自然升起。
- 鼓和 bass 支撑了主打句。

坏变化：

- 风格突然变成另一首歌。
- 人声被盖住。
- 进入莫名其妙的 drop。
- 情绪断裂。

处理方式：

- 在段落元标签里写清楚进入方式。
- 减少互相冲突的曲风词。
- 降低 weirdness。
- 排除不想要的 drop 或过度编曲。

## 36. 完整性补全

如果生成结果像半成品：

- 看是不是歌词太短。
- 看是不是结构标签太少。
- 看是否只给了 hook 却期待完整歌。
- 看尾奏是否没有 ending。
- 看 prompt 是否只写情绪，没有曲风和制作方向。

补全时可以：

- 选择完整结构模板。
- 增加 verse / pre-chorus / chorus。
- 给尾段明确 ending。
- 用 extend 补后半段。

## 37. 旋律偏移

旋律偏移常见于：

- 歌词行长变化太大。
- 句子节奏忽快忽慢。
- 段落标签太复杂。
- prompt 曲风冲突。
- weirdness 过高。
- 试图在一首歌里完成太多变化。

解决方式：

- 统一每行字数和停顿。
- 把重点句重复或变奏。
- 降低 weirdness。
- 提高 style influence。
- 只保留一个主曲风。
- 用已成立版本继续 extend。

## 38. 常见问题处理

### 生成结果太土

可能原因：

- 曲风词太泛。
- vocal 描述太空。
- 编曲元素太传统。
- 歌词像老套情歌。

处理：

- 加入更具体的曲风参考。
- 明确 production texture。
- 减少陈词滥调。
- 换更适合当前听众的 vocal。

### 生成结果太像模板

可能原因：

- prompt 只有常见形容词。
- 歌词缺少具体故事。
- 曲风库参考不明确。

处理：

- 增加具体画面和听众情绪。
- 用好歌词案例校准第一句。
- 参考曲风库，而不是凭空写“高级”。

### 副歌不够抓耳

可能原因：

- 第一句太长。
- 记忆点在句尾太靠后。
- 旋律没有重复。
- 关键词不够具体。

处理：

- 缩短前两句。
- 把最重要的词提前。
- 增加押韵或节奏回环。
- 重新生成 hook，不急着做完整歌。

### 前 5 秒留不住

可能原因：

- 前奏太长。
- 第一口人声晚。
- 第一字不清楚。
- 第一句歌词没锁定人群。
- 画面和曲风不匹配。

处理：

- 裁出更早进入的人声段。
- 改第一句歌词。
- 让旋律记忆点前置。
- 换画面模板或首屏文案钩子。

## 39. 高风险误区

- 以为 prompt 越长越专业。
- 以为 voice 能解决所有人声问题。
- 以为 cover 可以安全复刻现存歌曲。
- 以为 extend 能把坏片段救成好歌。
- 以为母带能解决旋律和歌词问题。
- 以为音频分离后就可以随便商用。
- 以为生成工具输出就是最终成品。
- 以为只要歌好听，短视频画面无所谓。

## 40. bmzskill 回答 Suno 问题的固定边界

回答 Suno 技术问题时必须做到：

1. 能给操作方向，但不编造当前界面按钮。
2. 能给排查顺序，但不承诺自动听音频判断。
3. 能给 prompt，但要说明是参考提示词。
4. 能给版权风险提醒，但不给法律结论。

## 42. 与 bmzskill 其他模块的关系

- `/bmz-lyrics`：负责歌词、结构、主打段，不负责平台按钮。
- `/bmz-prompt`：负责把歌词、曲风、vocal、编曲和参数整理成可用 prompt。
- `/bmz-style`：负责从曲风库推荐参考方向。
- `/bmz-preflight`：负责发布前判断画面、歌曲、歌词、prompt 是否适配。
- `/bmz-review`：负责看发布后数据，不用生成工具结果自嗨。
- `/bmz-ask`：负责解释 Suno 操作问题和技术排查路径。
