# 歌曲短视频发布闭环工作流

本工作流用于帮助 `/bmz` 推进一首原创音乐 demo 从主题到发布复盘的完整闭环。它不是公开入口，而是内部路线图。

## 总目标

把用户的一首歌推进成一次可复盘的市场测试：

```text
主题假设 -> 创作启发 -> 歌词质检 -> 发布前体检 -> 发布包生成 -> 发布后复盘 -> 下一轮决策
```

每一步都要服务同一个问题：这首歌是否值得继续测试、修改、发布或放大。

## 角色分工

- `/bmz/SKILL.md`：负责识别用户当前状态、选择内部流程、统一回复。
- `templates/songwriting.md`：用户只有主题或想法时，用来拆听众、方向和 8 句主打段结构。
- `templates/lyrics-check.md`：用户有歌词或主打段时，用来判断歌词风险。
- `templates/preflight.md`：用户有视频、截图、首帧或待发布素材时，用来判断发布前风险。
- `templates/publish.md`：用户准备发布时，用来生成标题、首屏文案钩子、置顶评论和标签。
- `templates/review.md`：用户发布后有数据时，用来判断下一轮动作。

## 阶段判断

### 阶段 1：只有主题

用户表现：

- 只有一句主题。
- 说“想写一首歌”。
- 还没有歌词或视频。

处理：

- 使用 `templates/songwriting.md`。
- 不默认直接写完整歌词。
- 输出目标听众、3 个可写方向和 8 句主打段结构。

下一步：

```text
/bmz 检查我的 8 句主打段
```

### 阶段 2：已有歌词草稿

用户表现：

- 发来歌词、hook、主打段或几句词。
- 想知道这段能不能用。

处理：

- 使用 `templates/lyrics-check.md`。
- 默认只审查，不主动改写。
- 重点判断第一句抓人、活人感、逻辑、情绪递进和短视频传播适配。

下一步：

```text
/bmz 复查我修改后的歌词
```

如果歌词已经方向可用，可以进入：

```text
/bmz 检查这条待发布视频
```

### 阶段 3：已有待发布视频或首帧

用户表现：

- 上传视频、截图、封面、首帧。
- 给了标题、标签、歌词呈现或发布包草稿。

处理：

- 使用 `templates/preflight.md`。
- 不承诺专业音频分析。
- 判断画面、字幕、标题、标签、首屏文案钩子和歌词是否打向同一类听众。

下一步：

```text
/bmz 给这条作品做发布包
```

### 阶段 4：准备发布

用户表现：

- 问标题怎么写。
- 要首屏文案钩子、置顶评论、标签。
- 说“准备发了”。

处理：

- 使用 `templates/publish.md`。
- 发布包必须服务同一类听众。
- 标签不超过 5 个，必须包含 `#原创音乐` 和 `#demo`。

下一步：

```text
/bmz 复盘这条作品数据
```

### 阶段 5：已有发布数据

用户表现：

- 给后台截图。
- 给播放、2 秒跳出、5 秒留存、完播、点赞、收藏、评论、转发等数据。
- 问下一步该不该换画面、改 Hook、换歌、继续发或投流。

处理：

- 使用 `templates/review.md`。
- 判断问题发生在画面、歌词、发布包、分发人群还是歌曲资产。
- 不鼓励无脑投流。

下一步按结论选择：

```text
/bmz 复查我修改后的歌词
/bmz 检查第二轮版本
/bmz 重新做发布包
/bmz 开始下一首
```

## 闭环规则

- 不要让用户跳过歌词质检直接进入发布包，除非用户明确只需要发布包装。
- 发布前体检发现歌词问题明显时，先回到歌词质检。
- 发布后复盘发现 2 秒跳出高，优先回到发布前体检看首帧、画面和首屏文案钩子。
- 发布后复盘发现 5 秒留存低，优先回到歌词质检或主打段结构。
- 发布后复盘发现互动弱但留存尚可，优先回到发布包生成。
- 同一首歌或同一画面模板最多建议两轮测试。两轮后仍无有效信号，应建议换方向。

## 回复习惯

每次回复都要让用户知道：

1. 当前处在哪个内部流程。
2. 这一步要解决什么判断问题。
3. 当前最主要风险是什么。
4. 下一步应该做什么。

不要让用户自己选择模板，不要暴露文件路径，不要把内部模板名当作公开入口。
